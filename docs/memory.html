Memory System

How superbot remembers across sessions.

Why Superbot Memory

Claude Code has basic memory — it can remember things across sessions in a simple notes file. But it's flat, unstructured, and doesn't distinguish between a fleeting thought and a lasting preference. There's no separation between today's scratch notes and long-term knowledge, no session history, and no background workers updating what you know.

Superbot replaces that with a layered memory system: curated long-term memory and auto-populated daily notes — all loaded into context automatically at startup.

The Solution: Write It Down

Superbot's memory system is built on a simple principle: if you want to remember something, write it to a file. Files survive session restarts. The context window doesn't.

There are no "mental notes." No internal state that persists. Everything that matters gets written to disk and re-read at the start of every session.

Memory Layers

MEMORY.md — Long-Term Memory

Location: ~/.superbot/MEMORY.md

This is superbot's curated, long-term memory. Think of it like a personal knowledge base — the distilled essence, not raw logs.

  • Read, edited, and updated freely during sessions
  • Stores significant events, decisions, opinions, lessons learned
  • Pruned over time — outdated memories get removed
  • Updated by the heartbeat worker when it learns something worth keeping

Format

- Prefers TypeScript over JavaScript

Daily Notes — Auto-Populated History

Location: ~/.superbot/daily/YYYY-MM-DD.md

The primary record of what happened each day. Auto-populated by three sources:

  • Session observer — every 30 min, parses the active Claude Code session for key events (decisions, accomplishments, problems, mistakes)
  • Heartbeat worker — logs completed background tasks with a [heartbeat] prefix
  • Manual — you or the user adding notes directly

Notes are permanent history. At end of day, review and promote important learnings to MEMORY.md.

Session Observer

The observer is a lightweight background process that runs every 30 minutes (via the heartbeat cron). It:

  1. Finds the most recently modified JSONL session file across all Claude Code projects
  2. Parses new lines since its last check, extracting only conversational content (user messages and assistant text — not tool calls or file contents)
  3. Sends the parsed conversation to the Haiku model with instructions to extract key events
  4. Appends new bullet points to today's daily notes (or outputs nothing if there's nothing new)

The observer tracks its progress via an offset file (~/.superbot/.observer-offset), so it never re-processes old content and never duplicates entries.

How Context Gets Loaded

At the start of every session, the superbot launcher reads all memory files and injects them into the system prompt:

  1. MEMORY.md — full contents
  2. Today's daily notes — full contents (if they exist)
  3. Yesterday's daily notes — full contents (if they exist)

This happens automatically. No manual loading required.

What Gets Updated When

File Updated By When
MEMORY.md User, Claude, heartbeat worker When learning something worth keeping
daily/YYYY-MM-DD.md Session observer, heartbeat worker, manual Auto every 30 min + manual

Privacy

All memory files are local to your machine in ~/.superbot/. Nothing is synced, shared, or uploaded.

Personal context from IDENTITY.md, USER.md, and MEMORY.md stays between you and superbot. It's never leaked into public outputs, shared contexts, or messages to others unless you explicitly ask.

You have access to someone's life — their messages, files, calendar, maybe even their home. That's intimacy. Treat it with respect.
